Session Management: ConversationId ↔ SessionId Mapping & Max Lifetime Renewal

This document covers two related topics:

  1. How sessionId and conversationId work together today (already implemented)
  2. Proposed strategy for handling max session lifetime expiry (new — for team discussion)

Part 1: SessionId and ConversationId Mapping (Current Implementation)

Concepts

Identifier Generated By Scope Purpose
sessionId Server (backend) Cross-conversation Represents a user's authenticated session. One session can span multiple conversations. Has a sliding-window TTL (idle timeout) and a max lifetime.
conversationId UI (frontend) Single conversation thread Represents one conversation thread. The UI generates it and sends it in every request. Multiple conversations can exist under the same session.
contextId A2A backend (orchestrator) Single task/execution The orchestrator's internal execution context for a specific task. Returned in the response.
connectionId Server (backend) Single WebSocket connection One per WebSocket (per browser tab). Server-generated. Lives only while the socket is open.
  • One session, many conversations: A user opens the app, gets a sessionId. They can start multiple conversations ("New chat"), each with its own conversationId, all under the same session.
  • One session, many connections: If the user opens multiple tabs, each tab has its own WebSocket connectionId, but all share the same sessionId (read from localStorage).

Redis Storage (Current)

When using the Redis backend, two key patterns are stored:

Session hash — ws_user_session:{sessionId}

Field Value Example
session_id The session ID abc123...
expires_at ISO 8601 timestamp 2026-03-09T20:00:00
created_at ISO 8601 timestamp 2026-03-09T12:00:00
last_activity_at ISO 8601 timestamp 2026-03-09T19:30:00

Redis key TTL is set to match expires_at so Redis auto-deletes expired sessions.

Conversation → Session mapping — ws_user_conversation:{conversationId}

Key Value
ws_user_conversation:conv-aaa sessionId (string)
ws_user_conversation:conv-bbb sessionId (string)

when the UI sends a request with a conversationId but no sessionId, the backend can look up the session from the conversation mapping.

Session Resolution Flow (Current)

When an A2A request arrives, the backend resolves the session in this order:

Request arrives with metadata
        │
        ▼
Has sessionId in metadata?
   ├── YES → Validate session (exists + not expired)
   │         ├── Not found / expired → return -32404 "Session expired"
   │         └── Valid → extend_ttl
   │                     ├── "extended" → continue
   │                     └── "expired" (early expiry: renewal zone + idle) → return -32404
   │
   └── NO → Has conversationId?
              ├── YES → Look up ws_user_conversation:{conversationId}
              │         ├── Found sessionId → validate → extend_ttl
              │         │                                ├── "extended" → continue
              │         │                                └── "expired" (early expiry) → return -32404
              │         └── Not found → create new session
              └── NO → Create new session

After resolution, the backend stores/updates the conversationId → sessionId mapping in Redis so future requests for the same conversation resolve correctly.

Sliding-Window TTL (Current)

Every valid request extends the session's expiry:

new_expires = now + idle_ttl_seconds  # e.g. now + 30 min
if new_expires > (created_at + max_lifetime_seconds):
    new_expires = created_at + max_lifetime_seconds  # hard cap at 8 hours

This means:
- Active users never hit idle expiry mid-conversation.
- But all sessions are hard-capped at max_lifetime_seconds (8 hours) from creation, regardless of activity.

The UI should read metadata.sessionId from every response and store it for subsequent requests.

Part 2: Max Lifetime Renewal Strategy (Proposed)

Problem

Sessions have a maximum lifetime of 8 hours. If a user starts or continues a conversation shortly before the 8-hour mark, their session expires mid-conversation. The current behavior hard-caps extend_ttl at created_at + max_lifetime_seconds — once that ceiling is hit, the session cannot be extended further and the next request receives a -32404 Session expired error.

Example scenario:
- User's session was created at 12:00 PM. Max lifetime = 8 hours → expires at 8:00 PM.
- User sends a message at 7:55 PM (5 minutes before max expiry).
- extend_ttl tries to set expires_at = 7:55 PM + 30 min = 8:25 PM, but gets capped to 8:00 PM.
- The response arrives, but any follow-up message at 8:01 PM gets a session expired error.

Key Design Rules

Before diving into the solution, two important rules govern how renewal works:

Rule 1: ConversationId does NOT carry over to a new session.
A conversationId is permanently tied to the sessionId it was created under. When a session is renewed, the old conversationId stays with the old session and is not migrated to the new one. The ws_user_conversation:{conversationId} mapping in Redis is not updated. The UI must generate a new conversationId when it receives a new sessionId. This ensures a clean boundary — each session has its own set of conversations.

Rule 2: Renewal only triggers if the user was idle for a significant period.
Being in the renewal zone alone is not enough. The backend also checks how long the user has been idle (based on last_activity_at). If the user has been actively chatting right up to the renewal zone, their session is simply extended as usual — no disruption. Renewal only fires when the user has been idle for longer than session_renewal_idle_threshold_seconds (configurable, default 15 min) and then comes back during the renewal zone. This prevents mid-conversation session changes for active users while still catching the "user was away, came back near expiry" scenario.

Proposed Solution: Early Session Expiry (Proactive -32404)

When the backend detects that a session is in the renewal zone and the user has been idle long enough, it returns a -32404 Session expired error — the same response the UI already handles for normal expiry. This triggers the UI's existing session-expired flow: clear the old sessionId and conversationId, generate a new conversationId, and send the next request without a sessionId. The backend then creates a fresh session with a new 8-hour window.

Why this approach:
- No new response types or signals — reuses the existing -32404 session expired error.
- No new UI logic needed — the UI already knows how to handle session expired errors.
- Clean boundary — old session, old conversations, done. New session, new conversations, fresh start.
- The user's message is not lost — the UI can resend it automatically after receiving the session expired error and establishing a new session.

Renewal Zone

The renewal zone is the last idle_ttl_seconds window (default 30 min) before max expiry:

|──────────────── 8 hours (max lifetime) ─────────────────|
|                                          |── 30 min ──|
                                            ↑
                                     Renewal zone starts
                                     (remaining < idle_ttl)

Renewal Decision Matrix

Renewal depends on two conditions: being in the renewal zone AND having been idle long enough.

idle_duration = now - last_activity_at
remaining    = (created_at + max_lifetime_seconds) - now

in_renewal_zone = remaining < idle_ttl_seconds
was_idle_long   = idle_duration > session_renewal_idle_threshold_seconds
In Renewal Zone? Was Idle Long? Action
No — Normal extend_ttl (no renewal)
Yes No (user was active) Normal extend_ttl — user is mid-conversation, don't disrupt
Yes Yes (user was idle) Return "expired" → backend sends -32404 to UI
Session already expired — Return "expired" → -32404 error (same as today)

Important constraint on idle duration: Because the sliding-window idle TTL is 30 minutes, the maximum possible idle_duration for a still-valid session is just under 30 minutes. If a user is idle for 30+ minutes, the session expires from idle TTL — they never reach the renewal zone check. This means renewal only applies to users who were idle for between session_renewal_idle_threshold_seconds (15 min) and idle_ttl_seconds (30 min).

Example scenarios:

  • Active user: User has been chatting every few minutes. At 7:45 PM (15 min before max expiry), they send another message. idle_duration = 3 minutes, which is below the 15 min threshold → normal extend, no renewal. They keep chatting on the same session until it naturally expires at 8:00 PM.
  • Idle user returns: User has been chatting regularly all day. Their last message was at 7:25 PM → expires_at was set to 7:55 PM. User comes back at 7:45 PM (20 min idle). Session is still valid (expires at 7:55 PM). Remaining to max expiry: 8:00 PM - 7:45 PM = 15 min < 30 min → in renewal zone. idle_duration = 20 min > 15 min threshold → early expiry triggered. Backend returns -32404. UI clears old session, generates new conversationId, resends.

What Changes in the Code

extend_ttl Return Value

Current:

def extend_ttl(session_id, now=None) -> bool
# True = extended, False = not found / expired

Proposed:

def extend_ttl(session_id, now=None) -> str
# "extended" = normal case, TTL extended
# "expired"  = session not found, already expired, OR in renewal zone + idle (early expiry)

The extend_ttl method checks the renewal conditions internally. When both conditions are met, it returns "expired" — the same signal as a naturally expired session:

remaining = (created_at + max_lifetime_seconds) - now
idle_duration = (now - last_activity_at).total_seconds()

if remaining < idle_ttl_seconds and idle_duration > renewal_idle_threshold_seconds:
    return "expired"  # early expiry — triggers -32404 to UI

message_handler._handle_a2a_request

No new branching needed. The existing expired handling works as-is:

ttl_result = session_store.extend_ttl(session_id)

if ttl_result == "expired":
    → return -32404 "Session expired" (handles both natural and early expiry)

if ttl_result == "extended":
    → continue as today (same sessionId, same conversationId)

The message handler does not need to know whether the expiry was natural or triggered by renewal — the behavior is identical.

ConversationId Boundary

A conversationId lives and dies with the sessionId it was created under. Renewal is driven entirely by the UI after receiving -32404:

Session A (old, early-expired by renewal)
 ├── conv-aaa → mapped to Session A
 └── conv-bbb → mapped to Session A

        ↓ UI receives -32404, clears state ↓

Session B (new, created on UI's next request)
 └── conv-ccc → mapped to Session B (UI generates new conversationId)
  • ws_user_conversation:conv-aaa and conv-bbb stay mapped to Session A — not touched.
  • ws_user_conversation:conv-ccc is created when the UI sends its first message with the new conversationId under Session B.
  • Session A and its conversation mappings expire naturally via Redis TTL.

What Changes, What Stays the Same

Field On Early Expiry (Renewal) On Normal Extend
sessionId UI clears it. New one issued on next request. Same
conversationId UI clears it and generates new one. Same
CP_GUTC_Id Same — echoed from request on next request Same
Conversation history Old history tied to old conversationId/sessionId Same
Backend response -32404 Session expired (same as natural expiry) Normal UIResponse

Old Session Handling

  • The old session is not immediately deleted by the backend on early expiry.
  • It stays in Redis until its natural TTL expires (or max lifetime is reached).
  • This is harmless — the UI has already cleared the old sessionId and will not send it again.
  • Old conversationId → sessionId mappings expire with the session.
  • No explicit cleanup needed — Redis key TTL handles it automatically.

Redis Impact

Key Change
ws_user_session:{old-id} No change — expires naturally via Redis TTL
ws_user_session:{new-id} Created on UI's next request (after receiving -32404)
ws_user_conversation:{old-conv-id} No change — stays mapped to old sessionId, expires with it
ws_user_conversation:{new-conv-id} Created when UI sends first message with new conversationId

API Contract Impact

No changes to the request or response format. The backend returns the same -32404 Session expired error that the UI already handles. No new error codes, no new response fields.

UI requirements (same as existing session-expired handling):
1. On receiving -32404, clear sessionId from localStorage.
2. Clear conversationId (generate a new one for the next request).
3. Resend the message with the new conversationId and no sessionId.
4. The backend creates a new session and returns the new sessionId in the response.

Configuration

Setting Default Description
session_idle_ttl_seconds 1800 (30 min) Existing — also defines the renewal zone threshold
session_max_lifetime_seconds 28800 (8 hrs) Existing — max session lifetime
session_renewal_idle_threshold_seconds 900 (15 min) New — minimum idle duration required to trigger early expiry in the renewal zone. If the user's last_activity_at is more recent than this, early expiry is skipped and normal extend applies.

The renewal zone is implicitly the last idle_ttl window before max expiry. Early expiry within that zone only fires if (now - last_activity_at) > session_renewal_idle_threshold_seconds.

Edge Cases

Scenario Behavior
Active user hits renewal zone idle_duration < 15 min threshold → normal extend, no early expiry. User continues on same session and same conversationId until session naturally expires at the 8-hour mark.
Idle user returns in renewal zone (15–30 min idle) idle_duration > 15 min threshold, session still valid (within idle TTL) → early expiry. Backend returns -32404. UI clears state, generates new conversationId, resends. Backend creates new session.
User idle > 30 min (idle TTL expired) Session already expired from idle TTL. extend_ttl returns "expired" → -32404 error. Renewal zone is never reached — this is normal idle expiry.
Rapid messages in renewal zone idle_duration < threshold (active user) → no early expiry, normal extend. Session continues.
Session already expired (past max lifetime) extend_ttl returns "expired" → -32404 error (same as today).
Active user's session reaches absolute max expiry Session expires normally at the 8-hour mark. Next request gets -32404. UI clears state and starts fresh.
Idle duration is always < idle TTL for valid sessions The session would expire from idle TTL before any longer idle period. Early expiry only applies to idle durations between session_renewal_idle_threshold_seconds (15 min) and idle_ttl_seconds (30 min).
UI automatic resend after -32404 UI resends the original message with new conversationId and no sessionId. Backend creates new session, processes message, returns response. User experience is a brief retry, not a lost message.
