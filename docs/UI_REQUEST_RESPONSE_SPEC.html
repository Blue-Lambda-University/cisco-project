UI Request and Response – Data Contract

This document defines the WebSocket payload contract between the UI and the backend: every field in the request and response, whether it is required or optional, and sample payloads. All communication is over a single WebSocket connection; the UI sends JSON requests and receives JSON responses.

1. High-level flow

  1. UI opens a WebSocket to the backend (e.g. ws://host/ciscoua/api/v1/ws). Subprotocol (e.g. cdca2a) is optional.
  2. UI sends one JSON request per user action. The request has top-level jsonrpc, id, and params (see Request Payload below).
  3. Backend validates the request, resolves or creates a session, and returns exactly one of: a success result, an error, or an in-progress result (with a final result sent later on the same WebSocket).
  4. UI uses the response: stores result.metadata.sessionId and result.metadata.conversationId (or result.contextId) for follow-up; replaces {user_name} in welcome text when applicable.

2. Request payload – field reference

The UI sends a single JSON object per message. All field names are camelCase in the wire format.

2.1 Top-level

FieldRequiredTypeDescription
jsonrpcYesstringMust be "2.0".
idNostring | numberRequest identifier. Echoed in the response. Recommended: string (e.g. "req-a1b2c3d4").
paramsYesobjectMust contain message; may contain metadata. See below.

2.2 params.message

FieldRequiredTypeDescription
roleNostringSender role. Default "user".
partsYes*arrayContent parts. Each element has kind (optional, default "text") and text (string). For a normal query, at least one part with non-empty text is required. *Exception: when params.metadata.isFirstChat is true, parts may be empty (backend returns first-chat welcome).
messageIdNostringClient message id.
contextIdNostringConversation/context id for follow-up turns. Should match the conversationId the UI is using for this thread.

parts[] element:

FieldRequiredTypeDescription
kindNostringPart type. Default "text".
textYes (for text parts)stringUser query or content.

2.3 params.metadata

Entire object is optional. If present, all fields inside are optional.

FieldRequiredTypeDescription
sessionIdNostring | nullSession id from a previous response. Omit or null for first turn; backend creates one and returns it in result.metadata.sessionId.
conversationIdNostringUI-generated conversation id. Backend echoes it in the response. Use the same value for all messages in the same conversation.
CP_GUTC_IdNostringPass-through; echoed in result.metadata.
referrerNostringPass-through; echoed in result.metadata.
isFirstChatNobooleantrue: backend returns the first-chat welcome message (query in parts may be empty). false or omit: normal query; at least one part with non-empty text is required.

Clarifications:

  • sessionId: For follow-up in the same session, send the value from result.metadata.sessionId of the previous response. If the backend returns an error with code -32000 (session expired/not found), clear stored sessionId and send the next request without it (or with null).
  • conversationId: Always generated by the UI. Backend never creates it; it only echoes it. Use the same conversationId for all messages in the same thread.
  • isFirstChat: Set to true only on the very first message of a new conversation to get the welcome message. Set to false or omit for normal queries.

3. Response payload – field reference

The backend sends a single JSON object per response. Either result is present (success or in-progress) or error is present (error). All field names are camelCase.

3.1 Top-level

FieldRequiredWhenTypeDescription
jsonrpcYesAlwaysstring"2.0".
idNoAlwaysstring | number | nullEcho of the request id.
resultYes (on success/in-progress)When no errorobjectTask result; see below.
errorYes (on error)When errorobjectcode and message. See Error response.

3.2 Success response: result

FieldRequiredTypeDescription
kindYesstring"task".
idYesstringTask id from backend.
contextIdNostring | nullEcho of request conversationId (or null).
statusYesobjectstate, optional message, timestamp.
artifactsYesarrayAt least one artifact; each has artifactId, name, parts. Response content is in artifacts[].parts[].text.
roleYesstring"assistant".
metadataNoobjectSee result.metadata below.

result.status:

FieldRequiredTypeDescription
stateYesstringe.g. "completed", "in_progress".
messageNostringOptional status message.
timestampNostringISO 8601.

result.artifacts[]:

FieldRequiredTypeDescription
artifactIdYesstringId of the artifact.
nameNostringHuman-readable name.
partsYesarrayContent parts. Each has kind (e.g. "text") and text (response content).

result.metadata (optional; backend typically populates for session/conversation continuity):

FieldRequiredTypeDescription
timestampNostringISO 8601 response time.
sessionIdNostring | nullSession id. UI should store this and send it in the next request for the same session.
conversationIdNostring | nullEcho of request conversationId. Use for follow-up.
CP_GUTC_IdNostring | nullEcho of request value.
referrerNostring | nullEcho of request value.

Where to read session/conversation for follow-up: Use result.metadata.sessionId and result.metadata.conversationId (or result.contextId, which matches conversationId). Session id is only inside result.metadata, not at the top level of result.

3.3 Error response: error

FieldRequiredTypeDescription
codeYesnumbere.g. -32602 (invalid params / missing or empty query), -32000 (session expired or not found).
messageYesstringHuman-readable message.

When the UI receives an error with code -32000, it should clear any stored sessionId and treat the next user action as a new session (omit or null sessionId in the next request).

3.4 In-progress response

The backend may respond with result.status.state = "in_progress" and an optional result.status.message. The final result for that request is then sent later on the same WebSocket. The UI should treat this as “processing” and wait for the final result on the same connection.

4. First chat vs normal query

ScenarioUI sends isFirstChatUI sends partsBackend behavior
First chattrueMay be emptyReturns welcome message: "Welcome {user_name}! I am Cisco Uber Assistant. How can I help you today?" plus action lines. UI replaces {user_name}.
Normal queryfalse or omitAt least one part with non-empty textProcesses query; returns content or a fallback message in result.artifacts.
Empty query, not first chatfalse or omitEmptyBackend returns error -32602 (missing/empty query).

5. Sample payloads

5.1 Request – first turn

{
  "jsonrpc": "2.0",
  "id": "req-a1b2c3d4",
  "params": {
    "message": {
      "role": "user",
      "parts": [
        { "kind": "text", "text": "get cases for test@cisco.com" }
      ],
      "messageId": "msg-001"
    },
    "metadata": {
      "sessionId": null,
      "conversationId": "conv-uuid-1111-2222-3333",
      "CP_GUTC_Id": "gutc-abc123",
      "referrer": "https://support.example.com",
      "isFirstChat": true
    }
  }
}

5.2 Request – follow-up

Use sessionId and conversationId from the previous response. Send the same conversationId in both params.metadata.conversationId and params.message.contextId.

{
  "jsonrpc": "2.0",
  "id": "req-e5f6g7h8",
  "params": {
    "message": {
      "role": "user",
      "contextId": "conv-uuid-1111-2222-3333",
      "parts": [
        { "kind": "text", "text": "show me the first one" }
      ],
      "messageId": "msg-002"
    },
    "metadata": {
      "sessionId": "39HARr8dJYEUEeFDYheE4d3-q3EsSlPO7CBHyd_4YSI",
      "conversationId": "conv-uuid-1111-2222-3333",
      "CP_GUTC_Id": "gutc-abc123",
      "referrer": "https://support.example.com",
      "isFirstChat": false
    }
  }
}

5.3 Response – success

{
  "jsonrpc": "2.0",
  "id": "req-a1b2c3d4",
  "result": {
    "kind": "task",
    "id": "task_xyz-789",
    "contextId": "conv-uuid-1111-2222-3333",
    "status": {
      "state": "completed",
      "message": null,
      "timestamp": "2025-02-18T12:00:00.000000+00:00"
    },
    "artifacts": [
      {
        "artifactId": "art-001",
        "name": "agent_response",
        "parts": [
          { "kind": "text", "text": "Here are your cases: ..." }
        ]
      }
    ],
    "role": "assistant",
    "metadata": {
      "timestamp": "2025-02-18T12:00:00.000000+00:00",
      "sessionId": "39HARr8dJYEUEeFDYheE4d3-q3EsSlPO7CBHyd_4YSI",
      "conversationId": "conv-uuid-1111-2222-3333",
      "CP_GUTC_Id": "gutc-abc123",
      "referrer": "https://support.example.com"
    }
  }
}

5.4 Response – error (invalid params)

{
  "jsonrpc": "2.0",
  "id": "req-a1b2c3d4",
  "error": {
    "code": -32602,
    "message": "Invalid params: expected params.message with role and parts (e.g. parts[].text)."
  }
}

5.5 Response – error (session expired)

{
  "jsonrpc": "2.0",
  "id": "req-e5f6g7h8",
  "error": {
    "code": -32000,
    "message": "Session expired or not found."
  }
}

5.6 Response – in progress

The backend may send this when the work is deferred; the final result is sent later on the same WebSocket.

{
  "jsonrpc": "2.0",
  "id": "req-a1b2c3d4",
  "result": {
    "kind": "task",
    "id": "task-pending-abc",
    "status": {
      "state": "in_progress",
      "message": "Processing; final result will follow."
    }
  }
}

6. UI checklist

Every request:

  • Send valid JSON with jsonrpc: "2.0" and params containing message (with parts array).
  • For non–first-chat messages, ensure params.message.parts has at least one part with non-empty text, or the backend returns error -32602.
  • Use a stable id (e.g. string) if you need to correlate responses.

Session continuity (follow-up):

  • After the first successful response, store result.metadata.sessionId.
  • Send that value in params.metadata.sessionId on subsequent requests in the same session.
  • If the backend returns error -32000, clear stored sessionId and start a new session (omit or null sessionId on the next request).

Conversation threading:

  • Generate and store a conversationId (e.g. UUID) per conversation.
  • Send it in params.metadata.conversationId and, for follow-up, in params.message.contextId.
  • Use the same conversationId for all messages in that thread.

First-chat welcome:

  • On the very first message of a new conversation, set params.metadata.isFirstChat to true to receive the welcome message; replace {user_name} in the returned text with the actual user name.
