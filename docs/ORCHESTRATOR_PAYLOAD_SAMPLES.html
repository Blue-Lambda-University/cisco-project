Orchestrator Payload Samples

This document shows the exact JSON payloads exchanged between our WebSocket backend and the orchestrator for both the async (webhook) and sync (useOrchestrator) flows.

Correlation key: The requestId (from the UI's JSON-RPC id field) is used to correlate async webhook responses back to the correct WebSocket connection. It is included in the metadata sent to the orchestrator, echoed back in the webhook response, and used as the lookup key in the correlation store. If the UI does not provide a requestId, the backend generates a fallback UUID.

Architecture Overview

UI <──WebSocket──> Our Backend <──HTTP──> Orchestrator The WebSocket connection is ONLY between the UI and our backend. The orchestrator never touches the WebSocket — it only speaks HTTP.

Async Flow (Webhook)

UI ──WebSocket──> Our Backend ──POST /async/request──> Orchestrator │ │ │ (stores connection_id │ │ in correlation store │ (processes async) │ keyed by requestId) │ │ │ │ POST /webhooks/async/response │ │ <──────────────────────────────────────┘ │ (requestId in metadata) │ UI <──WebSocket────────┘ (looked up via requestId → connection_id)

Sync Flow (useOrchestrator: true)

UI ──WebSocket──> Our Backend ──POST /api/chat──> Orchestrator │ HTTP response (inline) │ UI <──WebSocket──── Our Backend <──────────────────────┘

1. Async Flow — What We Send to the Orchestrator

ASYNC → OUTGOING

Sent via POST {agent_base_url}/async/request with our webhook URL and the requestId embedded in metadata so the orchestrator echoes it back.

FieldTypeDescription
webhookUrlstringOur callback URL for async response delivery
messageobjectUser message (role, parts)
requestIdstringJSON-RPC request ID from the UI — used as the correlation key. If null, backend generates a UUID fallback.
sessionIdstringSession ID
contextIdstringConversation/context ID from the UI
CP_GUTC_IdstringCP GUTC ID from the UI
referrerstringReferrer from the UI

Sample Payload

{
  "webhookUrl": "https://our-service.example.com/ciscoua/api/v1/webhooks/async/response",
  "message": {
    "role": "user",
    "parts": [{ "kind": "text", "text": "get my licensing cases" }]
  },
  "requestId": "req-001",
  "sessionId": "xK9mN2pQ4rStUvWx...",
  "contextId": "conv-abc-456",
  "CP_GUTC_Id": "gutc-user-789",
  "referrer": "cisco.com/support"
}
Correlation store entry (backend-internal):
Key: "req-001"
Value: { connection_id: "conn-xxx", session_id: "xK9mN2pQ...", conversation_id: "conv-abc-456", query_text: "get my licensing cases", ... }

2. Async Flow — What the Orchestrator Sends Back (Webhook)

ASYNC ← INCOMING

The orchestrator POSTs to our webhook endpoint:

POST {webhook_base_url}/ciscoua/api/v1/webhooks/async/response

The requestId is included in the response body (part of the metadata the orchestrator echoes back). Our webhook handler uses it to look up the correlation store and find the correct WebSocket connection.

FieldTypeDescription
requestIdstringEchoed from our outgoing request — used to look up the correlation store
contentstringThe orchestrator's response text
sessionIdstringEchoed from our outgoing request
contextIdstringEchoed from our outgoing request
CP_GUTC_IdstringEchoed from our outgoing request
referrerstringEchoed from our outgoing request

Sample Payload

{
  "requestId": "req-001",
  "content": "Here are your licensing cases:\n\n<table>...</table>",
  "sessionId": "xK9mN2pQ4rStUvWx...",
  "contextId": "conv-abc-456",
  "CP_GUTC_Id": "gutc-user-789",
  "referrer": "cisco.com/support"
}
Webhook handler lookup:
1. Read requestId from response body → "req-001"
2. Look up correlation store: "req-001" → { connection_id: "conn-xxx", ... }
3. Build UIResponse → send to WebSocket connection "conn-xxx"
4. Remove entry from correlation store (one-time use)

3. Sync Flow — What We Send to the Orchestrator

SYNC → OUTGOING

Sent via POST {agent_base_url}/api/chat. Uses the orchestrator's REST schema (snake_case). No correlation store needed — response is returned inline.

FieldTypeRequiredDescription
querystringYesThe user's message text
user_idstringYesUser identifier (we send CP_GUTC_Id)
conversation_idstringYesConversation ID from the UI
session_idstringYesSession ID
request_idstringYesJSON-RPC request ID from the UI
metadataobjectNoAdditional metadata (CP_GUTC_Id, referrer)

Sample Payload

{
  "query": "get my licensing cases",
  "user_id": "gutc-user-789",
  "conversation_id": "conv-abc-456",
  "session_id": "xK9mN2pQ4rStUvWx...",
  "request_id": "req-001",
  "metadata": {
    "CP_GUTC_Id": "gutc-user-789",
    "referrer": "cisco.com/support"
  }
}

4. Sync Flow — What the Orchestrator Returns (Inline HTTP Response)

SYNC ← INCOMING

Returned directly in the HTTP response body of POST /api/chat.

FieldTypeDescription
conversation_idstringEchoed from request
session_idstringEchoed from request
request_idstringEchoed from request
task_idstringServer-generated UUID for this task
responsestringThe agent's complete response text
statusstring"completed" on success, "error" on failure
metadataobjectContains routed_agent, user_id, live_agent_mode
sub_agent_a2a_responseobject | nullRaw A2A events from the sub-agent (if any)

Sample Payload

{
  "conversation_id": "conv-abc-456",
  "session_id": "xK9mN2pQ4rStUvWx...",
  "request_id": "req-001",
  "task_id": "d4e5f6a7-b8c9-4d0e-1f2a-3b4c5d6e7f8a",
  "response": "Here are your licensing cases:\n\n<table>...</table>",
  "status": "completed",
  "metadata": {
    "routed_agent": "licensing-agent",
    "user_id": "gutc-user-789",
    "live_agent_mode": false
  },
  "sub_agent_a2a_response": {
    "events": [
      {
        "kind": "status-update",
        "taskId": "sub-task-001",
        "status": {
          "state": "completed",
          "message": {
            "role": "agent",
            "parts": [{ "kind": "text", "text": "Here are your licensing cases..." }]
          }
        }
      }
    ]
  }
}
