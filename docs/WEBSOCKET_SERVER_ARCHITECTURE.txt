WebSocket Server Architecture
cdcai-microsvc-uber-assistant-frontend
========================================


1. SESSION HANDLING
-------------------

We manage user sessions with a sliding-window TTL. Every time a user sends a
message, the session expiry slides forward. Active users don't get logged out,
but idle sessions auto-expire.

How it works:

  - User sends requestType: "welcome"
    -> session_store.create(user_token, email_address, ccoid)
    -> generates a random session_id using secrets.token_urlsafe
    -> sets expires_at = now + 30 minutes (idle_ttl_seconds)
    -> stores user_token, email_address, ccoid alongside the session
    -> returns session_id to UI in the response

  - User sends subsequent message with sessionId
    -> session_store.get(session_id) - checks if it exists and isn't expired
    -> if expired or missing -> returns error "Session expired or not found"
    -> if valid -> session_store.extend_ttl(session_id)
       -> new expires_at = now + 30 minutes
       -> capped by max_lifetime_seconds (8 hours from creation)
    -> session.expires_at is read and sent to UI as session_expiration_time

  - User sends requestType: "extend_session"
    -> session_store.renew_session(session_id)
    -> resets created_at to now, sets expires_at to now + 8 hours
    -> returns new session_expiration_time to UI

  - User sends requestType: "end_session"
    -> session_store.delete_session(session_id)
    -> removes the session from Redis/memory
    -> returns confirmation to UI

Two backends (same interface):

  InMemorySessionStore
    - Storage: Python dict in process memory
    - Multi-pod: No (each pod has its own sessions)
    - Persistence: Lost on restart
    - Config: UA_WS_SESSION_PERSISTENCE_BACKEND=memory

  RedisSessionStore
    - Storage: Redis hash per session
    - Multi-pod: Yes (shared across pods)
    - Persistence: Survives restart
    - Config: UA_WS_SESSION_PERSISTENCE_BACKEND=redis

Session fields:
  session_id        - str   - Random URL-safe token
  expires_at        - datetime - When the session expires (slides forward)
  created_at        - datetime - When the session was first created
  last_activity_at  - datetime - Last message timestamp
  user_token        - str   - Auth token from WebSocket header
  email_address     - str   - Email from WebSocket header
  ccoid             - str   - CCO ID from WebSocket header

Conversation mapping:
  A separate key ws_user_conversation:{conversationId} -> sessionId maps
  conversations to sessions so the same session can be reused across turns.

2. RATE THROTTLING
------------------

Prevents any single WebSocket connection from flooding the server. Uses a
token bucket algorithm -- allows bursts up to a limit, then throttles to a
steady rate.

How the token bucket works:
  - Bucket holds up to burst_size tokens (default: 5)
  - Tokens refill at rate_per_second (default: 10/min / 60 = 0.167/sec)
  - Each message consumes 1 token
  - If no tokens available -> message rejected with retryAfterMs

Practical example:
  - User can send 5 messages instantly (burst)
  - After that, roughly 1 message every 6 seconds
  - If they exceed the rate, they get a JSON-RPC error:
    {"jsonrpc":"2.0","id":null,"error":{"code":-32429,"message":"Rate limit exceeded","data":{"retryAfterMs":3200}}}

Scope: Per-connection only (not global). Each WebSocket connection gets its own
bucket instance.

3. REDIS
--------

Provides shared state across multiple pods for sessions and correlation tracking.

Connection setup:
  Both RedisSessionStore and RedisCorrelationStore use redis.asyncio (async
  client) with lazy initialization:

    import redis.asyncio as aioredis
    client = aioredis.from_url(redis_url, decode_responses=True,
        socket_connect_timeout=5, socket_timeout=5, retry_on_timeout=True)

Key namespaces:
  ws_user_session:{session_id}        Hash     Session data (expiry, user identity)
  ws_user_conversation:{conversationId} String  Maps conversationId -> sessionId
  ws_async_req:{request_id}           String   Tracks pending async requests (JSON)

Redis session hash fields:
  session_id       -> "abc123..."
  expires_at       -> "2026-03-25T18:19:38.855801"
  created_at       -> "2026-03-25T17:49:38.855801"
  last_activity_at -> "2026-03-25T17:49:38.855801"
  user_token       -> "Bearer eyJhbGciOi..."
  email_address    -> "user@cisco.com"
  ccoid            -> "8526b8c4-803c-49ff-a40f-defddc3976a0"
  TTL: auto-set via EXPIRE to match expires_at

Commands used (basic only -- compatible with all Redis versions):
  HSET, HGETALL, SET, GET, DELETE, EXPIRE, SCAN
  No advanced commands like GETDEL -- removed for compatibility with older
  Redis server versions.

5. HEADER DATA
--------------

We capture user identity headers from the UI's WebSocket handshake and forward
them as HTTP headers to the orchestrator. This keeps auth/identity out of the
JSON payload body.

Header mapping:

  Incoming (from UI)   Variable        Outgoing (to orch)   Stored in Redis
  user_token           user_token      X-User-Token         Yes
  email_address        email_address   X-User-Email         Yes
  ccoid                ccoid           X-User-ID            Yes

Extraction (once per connection):

  user_token    = websocket.headers.get("user_token", "")
  email_address = websocket.headers.get("email_address", "")
  ccoid         = websocket.headers.get("ccoid", "")

Extracted outside the message loop -- captured once at connection time, reused
for every message on that connection.

Forwarding to orchestrator:

  POST /a2a/ HTTP/1.1
  Host: orchestrator-svc:8006
  Content-Type: application/json
  X-User-Token: Bearer eyJhbGciOi...
  X-User-Email: user@cisco.com
  X-User-ID: 8526b8c4-803c-49ff-a40f-defddc3976a0

If any header is absent from the WebSocket connection, the corresponding
outgoing header is simply omitted -- no error.

6. CONNECTION MANAGEMENT
------------------------

Tracks all active WebSocket connections, enforces capacity limits, and provides
utilities for targeted messaging and broadcasting.

Capabilities:
  - Capacity control: max concurrent connections (default 1000). New ones
    rejected with WebSocket close code 1013.
  - Connection tracking: each connection gets a UUID, tracks client IP,
    connected_at, message_count, last_message_at.
  - Client IP detection: checks X-Forwarded-For header first (for load
    balancers), falls back to direct client IP.
  - Targeted send: send_to_connection(connection_id, text) pushes a message
    to a specific WebSocket by ID.
  - Broadcast: send to all connections or all subscribers of a topic.
  - Idle detection: get_idle_connections(timeout) returns connections that have
    been idle longer than the threshold.

8. IDLE CONNECTION TIMEOUT
--------------------------

idle too long.

Behavior:
  - Runs every timeout/6 seconds (minimum 30s)
  - Scans all connections for those idle longer than
    connection_idle_timeout_seconds (default 30 minutes)
  - Closes idle connections with WebSocket code 1000 and reason "Idle timeout"

9. ASYNC REQUEST TIMEOUT (CORRELATION SWEEP)
--------------------------------------------

Background task that sends timeout errors to the UI for orchestrator requests
that never received a response.

Behavior:
  - Runs every timeout/4 seconds
  - Scans the correlation store for requests older than
    async_response_timeout_seconds (default 30 minutes)
  - Sends a JSON-RPC error to the UI for each timed-out request:
    {"jsonrpc":"2.0","id":"<request_id>","error":{"code":-32408,"message":"Request timed out waiting for orchestrator response","data":{"timeoutSeconds":1800}}}

  3. Clears structlog logging context

    received content)
  - Skips intermediate streaming frames -- only the final accumulated
    response is sent to the UI
