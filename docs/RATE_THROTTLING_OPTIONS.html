WebSocket Message Rate Throttling — Algorithm Options

Problem

Once a WebSocket connection is established, a client can send unlimited messages at any rate. This can overwhelm the orchestrator, exhaust server resources, or degrade service for other users.

Goal

Limit how many messages a client can send per connection per time window. When exceeded, reject the message with an error but keep the connection open.

Option 1: Token Bucket I will go for this

How it works: Each connection gets a "bucket" with a fixed token capacity. Tokens refill at a steady rate. Each message consumes 1 token. When empty, messages are rejected until tokens refill.

Example: Bucket capacity = 5, refill rate = 1 token / 3 seconds (= 20/min)

TimeActionTokens
t=0sUser sends msg 14
t=0sUser sends msg 23
t=0sUser sends msg 32
t=0sUser sends msg 41
t=0sUser sends msg 50
t=0sUser sends msg 6REJECTED (wait ~3s)
t=3s1 token refills1
t=3sUser sends msg 70 (allowed)

Pros

  • Allows natural burst patterns (send a few quickly, then pause)
  • Smooth and predictable average rate
  • Simple implementation (~30 lines)
  • No external dependencies

Cons

  • Slightly more complex than fixed window

Option 2: Sliding Window

How it works: Track timestamps of all messages within a rolling time window (e.g., last 60 seconds). If the count exceeds the limit, reject.

Example: Max 20 messages per 60-second window

TimeMessages in last 60sAction
t=0s0Allowed
t=30s15Allowed
t=45s20Allowed
t=46s21REJECTED
t=61s20 (oldest dropped off)Allowed again

Pros

  • Very precise — no boundary burst issues
  • Accurate count at any point in time

Cons

  • Higher memory usage (stores every message timestamp)
  • More complex than token bucket
  • Overkill for per-connection rate limiting

Option 3: Fixed Window

How it works: Count messages in fixed time intervals (e.g., per minute). Reset counter at the start of each window.

Example: Max 20 messages per minute

TimeWindowCountAction
t=0:00Window 1 starts0
t=0:55Window 118Allowed
t=0:59Window 120Allowed
t=0:59Window 121REJECTED
t=1:00Window 2 starts0Counter reset
t=1:00Window 21Allowed

Pros

  • Simplest to implement
  • Lowest memory usage (just a counter + timestamp)

Cons

  • Boundary burst problem — client can send 20 msgs at t=0:59 and 20 more at t=1:00 (40 in 2 seconds)
  • Less fair/predictable than token bucket

Option 4: Leaky Bucket (FIFO Queue)

How it works: Incoming messages are placed in a queue. Messages are processed ("leaked") at a fixed rate. If the queue is full, new messages are dropped.

Example: Queue size = 5, processing rate = 1 message / 3 seconds

TimeActionQueue
t=0s5 messages arrive[1, 2, 3, 4, 5]
t=0s6th messageDROPPED (queue full)
t=3sMessage 1 processed[2, 3, 4, 5]
t=3sNew message accepted[2, 3, 4, 5, 6]

Pros

  • Very smooth, constant output rate
  • Good for protecting downstream systems

Cons

  • Adds latency (messages wait in queue)
  • More complex — changes the processing model
  • Users experience delayed responses
